As we can see above from our NMAP scan, we have 2 IPs 192.168.40.138 and 192.168.40.129 and they are configured inside domain mini.lab.

It is possible to see that we are dealing with a windows environment. Let’s configure FQDN.

Setting Up FQDN

A fully qualified domain name, sometimes also referred to as an absolute domain name, is a domain name that specifies its exact location in the tree hierarchy of the Domain Name System. It specifies all domain levels, including the top-level domain and the root zone.

Why do we configure an FQDN?

FQDNs are easier to remember than IP addresses and are needed to configure the DNS and IP address of a device on the internet.

For example, when trying to reach Google, it's much easier to type google.com in the browser, instead of finding and typing its numerical IP address. Getting an SSL certificate.

DNSMASQ

Dnsmasq is designed to act as a DNS forwarder, DHCP server, and TFTP server for small networks. You can use dnsmasq as an alternative to configuring separate DHCP and TFTP services.

First we start by installing dnsmasq

sudo apt install -y dnsmasq

Then we need to do some small configurations. We will make dnsmasq to use /etc/hosts file to read the machines configured on the file for the translation.

Let’s config the /etc/hosts file and add the IPs of the machines we need and their names.

sudo vim /etc/hosts

Now we edit the /etc/resolv.conf file and add the follow configuration.

sudo vim etc/resolv.conf

Last but not list we need to start our dnsmasq and make it read the /etc/hosts file and we are ready to go.

dnsmasq -H /etc/hosts

It’s all set, we can now test it by using host and the name of the servers and it will be translate it to the real IPs.

host dc.mini.lab

host ws.mini.lab

Setting Up Kerberos

We must install the Linux kerberos client

Questions should be answer in this order:

Insecure LSA Configuration Allowing Null Session Enumeration

An Insecure LSA Configuration Allowing Null Session Enumeration is a misconfiguration in Windows where the system allows unauthenticated users (null sessions) to request and receive information about domain users and groups. This happens because the Local Security Authority (LSA) is set to permit anonymous access, typically due to the RestrictAnonymous registry setting being set to 0. As a result, an attacker can connect without credentials and enumerate usernames and group memberships, which can be used for further attacks like password spraying or privilege escalation.

We can enumerate users using NetExec and if we have a misconfigured domain we will be able to retrieve the list of users and also their Description.

netexec smb dc.mini.lab -u '' -p '' --users

This Insecure LSA configuration allowed us to retried the list of some users registered in this domain, at least part of it. We found 2 non default users (Bob & Dave) and it is also possible to see that under Bob’s description we can find Bob’s password.

Before we try to login, we can also use NetExec to check if this user is really a valid user inside the domain.

Enumerating the mini.lab domain

Let’s start by running InvisiShell before we start our enumeration.

InvisiShell is a stealthy PowerShell execution framework that hides command execution from traditional logging and monitoring by hooking the .NET framework, allowing attackers to run PowerShell commands without generating typical event logs or command-line traces. We can simply execute the.bat file or copy/paste it inside our CMD session.

As soon as we run InvisiShell using the RunWithRegistryNonAdmin.bat script, it sets environment variables and registry keys that tell the .NET runtime to load a custom profiler DLL (InShellProf.dll) via COR_ENABLE_PROFILING and COR_PROFILER.

Once set, when we launch PowerShell, the .NET runtime loads that profiler DLL into the PowerShell process. This DLL hooks into the .NET methods, allowing our commands to be executed stealthily, without generating common PowerShell logs (like Event ID 4104, command-line arguments, or script block logs).

RunWithRegistryNonAdmin.bat

set COR_ENABLE_PROFILING=1
set COR_PROFILER={cf0d821e-299b-5307-a3d8-b283c03916db}

REG ADD "HKCU\Software\Classes\CLSID\{cf0d821e-299b-5307-a3d8-b283c03916db}" /f
REG ADD "HKCU\Software\Classes\CLSID\{cf0d821e-299b-5307-a3d8-b283c03916db}\InprocServer32" /f
REG ADD "HKCU\Software\Classes\CLSID\{cf0d821e-299b-5307-a3d8-b283c03916db}\InprocServer32" /ve /t REG_SZ /d "%~dp0InShellProf.dll" /f

powershell

set COR_ENABLE_PROFILING=
set COR_PROFILER=
REG DELETE "HKCU\Software\Classes\CLSID\{cf0d821e-299b-5307-a3d8-b283c03916db}" /f

In short: PowerShell starts in a "ghost mode," where it can execute commands without being logged by default security mechanisms.

Why We Saw Only 2 Users with Insecure LSA Configuration Allowing Null Session Enumeration, but More with Authenticated Access

When we performed enumeration with a null session (no credentials), we’re limited to only what anonymous users are allowed to see. This depends on how the domain is configured usually, anonymous access is heavily restricted, so we might only get a partial list or a few users that are exposed by default or misconfigured.

Once we compromise a valid domain user, we’re no longer anonymous now we're authenticated, and our compromised account has permissions to see more information in the domain like, Query Active Directory more fully, Get a complete list of domain users, Access objects that are protected from anonymous enumeration

Special Group

There is a special case here for a special group. If you remember from our Groups enumeration, there is a Group named ‘Enterprise Admins'

The Enterprise Admins group is a highly privileged security group in a Microsoft Active Directory (AD) forest. It exists only in the root domain of the AD forest and grants its members administrative privileges across the entire forest, including all child domains. Members of this group have the ability to manage any domain, Domain Controllers (DCs), and critical AD components across the forest.

From an offensive security perspective, enumerating the Enterprise Admins group is crucial because it provides insight into who holds the keys to the forest, opening pathways to achieve forest dominance.

Offensive Use Case Example

1. You compromise a low-privileged account in a child domain.

2. Enumerate Enterprise Admins to identify privileged accounts.

3. Use pass-the-hash, Kerberoasting, or other techniques to compromise an Enterprise Admin account.

4. With the Enterprise Admin account:

- Access the root domain.

- Perform a DCSync attack to dump the krbtgt hash, enabling a Golden Ticket attack.

- Use forest-level privileges to compromise all child domains or maintain persistence.

Takeaways

For a Red Teamer or Pentester:

- The Enterprise Admins group is the ultimate prize when targeting AD environments because of its forest-wide privileges.

- Enumerating this group provides a roadmap to identifying the most critical accounts in the environment.

- Once an Enterprise Admin is compromised, you essentially own the entire AD forest, enabling lateral movement, persistence, and complete control over the target network.

Get-ADGroupMember -Identity 'Enterprise Admins' -Server 'mini.lab'
