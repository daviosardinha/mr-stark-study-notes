    GOAD Overview
NTLM Relaying

We have now reached one of the most dynamic, fun and operationally devastating phases of network penetration: Network Poisoning and NTLM Relaying. This is where we move beyond simple information gathering and start actively manipulating network traffic to capture and weaponize authentication material. Instead of cracking passwords, we will relay them.

The core mechanics of this phase rely on exploiting legacy name resolution protocols. By positioning ourselves as a malicious man-in-the-middle, we can intercept authentication attempts and redirect them toward targets of our choosing. The success of relaying operations is entirely dictated by the configuration of our targets, specifically the enforcement of SMB Signing. This critical protocol feature, when enabled, cryptographically signs SMB communications to prevent exactly the type of tampering we intend to perform.

LLMNR/NBT-NS Poisoning and NTLM Stealing

Link-Local Multicast Name Resolution (LLMNR) and NetBIOS Name Service (NBT-NS) are legacy protocols that allow hosts to resolve names on the local network. When DNS fails to resolve a hostname, Windows falls back to these protocols, broadcasting queries to the local network. We can exploit this by responding to these broadcasts and claiming to be the requested host, tricking the victim into sending us their NTLM hash.

To execute this attack, we use Responder, a powerful tool that listens for LLMNR, NBT-NS, and MDNS queries and responds with our attacker's IP address.

Configure Responder

Before running Responder, we must modify its configuration to disable its own SMB and HTTP servers. This is crucial because we want to capture the NTLM hashes, not let Responder handle them. We need these protocols available for our relay tool (ntlmrelayx).

# Edit Responder.conf
SMB = Off
HTTP = Off

Initialize the Poisoner

We launch Responder on our primary interface, using the -w flag to start the WPAD rogue proxy server and -v for verbose mode to see exactly what's happening.

sudo responder -I eth0 -dw -v

As Responder runs, it listens for broadcast queries. When a victim mistypes a hostname or tries to access a non-existent share, they send out an LLMNR query. Responder immediately responds, telling the victim that we are the machine they're looking for. The victim then attempts to authenticate to us, sending their NTLM hash in the process.

NTLM Relay

While capturing hashes is useful, relaying them is where the real power lies. NTLM relay allows us to take the authentication attempt from one user and relay it to another machine, potentially gaining access without ever cracking a password.

However, NTLM relay only works against machines that do not enforce SMB Signing. During our initial reconnaissance, we identified that Castelblack, Braavos, and The-Eyrie all have SMB Signing disabled (signing:False), making them perfect targets for this attack.

1st - OPTION: Automated Secrets Dumping

This option uses ntlmrelayx to automatically dump SAM hashes, LSA secrets, and DPAPI data from relayed connections. We use the -tf flag to specify our targets file (hosts.txt), which contains the list of machines we identified earlier with SMB Signing disabled.

impacket-ntlmrelayx -tf hosts.txt -smb2support

When a victim authenticates to our poisoned response, ntlmrelayx captures their NTLM handshake and relays it to our target machines. If successful, it automatically dumps password hashes and other valuable secrets from the compromised systems.

Option 2: Interactive SMB Session Relaying

For more control, we can use the -i flag to establish an interactive SMB session. This creates a SOCKS proxy that we can connect to with smbclient or other tools, giving us interactive access to the relayed session.

impacket-ntlmrelayx -tf hosts.txt -smb2support -i
nc 127.0.0.1 11000

Option 3: SOCKS Proxy Pivoting

The most versatile option is SOCKS proxy pivoting. By using the -socks flag, ntlmrelayx maintains multiple relayed sessions and exposes them as SOCKS proxies. We can then use proxychains to route any tool through these compromised sessions.

impacket-ntlmrelayx -tf hosts.txt -socks -smb2support

IPv6 DNS Takeover

While most networks focus on IPv4, IPv6 is often left unconfigured but enabled. We can exploit this using mitm6, which acts as a rogue DHCPv6 server. When clients request IPv6 configuration, we provide them with our attacker's IP as their DNS server, allowing us to intercept and manipulate their DNS queries.

mitm6 -d sevenkingdoms.local

Coerced Authentication and Drop The Mic

Sometimes we need to force a victim to authenticate to us. We can use various coercion techniques like PetitPotam or PrinterBug to force a Domain Controller to authenticate to our malicious server, allowing us to capture or relay high-privilege credentials.

These techniques exploit legitimate Windows features to force authentication, making them difficult to detect and prevent.
