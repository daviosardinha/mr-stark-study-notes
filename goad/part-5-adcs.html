ADCS - ESC Attacks

Active Directory Certificate Services (ADCS) is Microsoft's Public Key Infrastructure (PKI) implementation. While designed to enhance security through certificate-based authentication, misconfigurations in ADCS can lead to devastating domain compromise scenarios. The infamous ESC (Exchange Server Certificates) attacks represent a family of vulnerabilities that allow attackers to escalate privileges and even achieve Domain Admin access.

Understanding ADCS attacks is crucial because certificates provide a stealthy persistence mechanism - they remain valid even if the user's password is changed, and they rarely trigger traditional security alerts.

How Does ADCS Work?

ADCS provides certificate-based authentication as an alternative to passwords. The process involves:

  • Certificate Authority (CA): The trusted entity that issues and manages certificates
  • Certificate Templates: Predefined configurations that determine how certificates are issued
  • Enrollment: The process of requesting and receiving a certificate

When properly configured, this system provides strong authentication. However, misconfigurations can allow attackers to request certificates for any user, including Domain Admins.

The Exploitation Logic of PKINIT

PKINIT is the Kerberos protocol extension that allows authentication using certificates instead of passwords. When a user presents a valid certificate, the Domain Controller issues a Ticket Granting Ticket (TGT) just as it would for a password-based authentication.

This means if we can obtain a certificate for a high-privilege account, we can request a TGT for that account and gain their privileges.

The ESC Families

ESC attacks are categorized based on the underlying vulnerability:

  • ESC1: Template allows requesters to specify SAN (Subject Alternative Name)
  • ESC2: Template allows "Any Purpose" EKU
  • ESC3: Enrollment Agent template allows certificate request on behalf of others
  • ESC4: Vulnerable Access Control on templates
  • ESC5: Vulnerable PKI Object Access Control
  • ESC6: EDITF_ATTRIBUTESUBJECTALTNAME2 flag enabled on CA
  • ESC7: Vulnerable Access Control on CA itself
  • ESC8: NTLM Relay to ADCS HTTP endpoint (Web Enrollment)

ESC1: Subject Alternative Name Spoofing

The most common and dangerous misconfiguration. When a certificate template allows the requester to specify their own Subject Alternative Name (SAN), we can request a certificate for any user, including Domain Admins.

The Operational Flow

To exploit ESC1, we need:

  1. A certificate template that allows Domain Users to enroll
  2. The template must have the "Supply in Request" option enabled for SAN
  3. The template must allow Client Authentication EKU

We use Certipy to enumerate vulnerable templates:

certipy find -u samwell.tarly -p 'Heartsbane' -target winterfell.north.sevenkingdoms.local -vulnerable

Once we identify a vulnerable template, we request a certificate for a high-privilege user:

certipy req -u samwell.tarly -p 'Heartsbane' -target winterfell.north.sevenkingdoms.local -ca 'sevenkingdoms-CA' -template 'VulnerableTemplate' -upn 'administrator@sevenkingdoms.local'

With the certificate in hand, we authenticate as the administrator:

certipy auth -pfx administrator.pfx -dc-ip 10.4.10.10

This provides us with the administrator's NT hash and a TGT, effectively giving us Domain Admin access.

ESC2: The Universal Key

ESC2 exploits templates with the "Any Purpose" Extended Key Usage (EKU). This allows the certificate to be used for any purpose, including client authentication, server authentication, code signing, etc.

The exploitation process is similar to ESC1, but we use a different template configuration.

ESC3: The Enrollment Agent

ESC3 exploits Enrollment Agent templates. These templates allow a user to request certificates on behalf of other users. If we can enroll in an Enrollment Agent template, we can request certificates for any user, including Domain Admins.

ESC4: Template Hijacking

ESC4 occurs when an attacker has write permissions to a certificate template. We can modify an existing template to make it vulnerable to ESC1, request a certificate, then restore the template to its original state.

ESC5: Vulnerable PKI Object

Similar to ESC4, but targets the CA itself or other PKI objects rather than templates. If we have write permissions to the CA object, we can enable the EDITF_ATTRIBUTESUBJECTALTNAME2 flag, making the CA vulnerable to SAN spoofing.

ESC6: CA Flag Manipulation

When the EDITF_ATTRIBUTESUBJECTALTNAME2 flag is enabled on a CA, it allows any requester to specify a SAN, even on templates that normally don't allow it. This effectively makes every template vulnerable to ESC1.

ESC8: NTLM Relay to ADCS

ADCS Web Enrollment interface accepts NTLM authentication. We can relay captured NTLM credentials to this endpoint to request certificates as the relayed user.

ntlmrelayx -t http://ca.sevenkingdoms.local/certsrv/certfnsh.asp -smb2support --adcs

Pass The Certificate

Once we have a certificate, we can use it for authentication in multiple ways:

Pass The Certificate with PKINIT

certipy auth -pfx administrator.pfx

Pass The Cert with Schannel

We can also use certificates with LDAPS for authenticated queries:

certipy auth -pfx administrator.pfx -ldap-shell

Prevention and Detection

To defend against ADCS attacks:

  • Regularly audit certificate templates for dangerous configurations
  • Monitor certificate enrollment logs for suspicious activity
  • Implement proper access controls on CA and template objects
  • Use dedicated CA administrators and avoid Domain Admin ownership
