Enterprise Admins Trust - SID-History Abuse with Trust Ticket

What is SID-History Abuse?

SID-History Abuse is a technique that allows an attacker to escalate privileges and bypass access controls in Windows systems. It involves injecting higher-privileged Security Identifiers (SIDs) into the SID History attribute of a user account, effectively granting the user the permissions associated with those higher-privileged groups.

How SID-History Abuse Works

  1. Compromising a Standard User Account: An attacker first compromises a standard user account with some level of privileges.
  1. Adding Higher-Privileged SIDs: The attacker then adds the SID of a higher-privileged group (such as Enterprise Admins or Domain Admins) to the SID History attribute of the compromised user account.
  1. Elevated Access: With the higher-privileged SIDs in the SID History, the user account now has elevated access to resources and systems within the parent domain.

SID-History Abuse using Trusted Tickets

The search results indicate that SID-History Abuse can be combined with the abuse of trusted tickets to further escalate privileges across domain boundaries:

  1. Compromising a Child Domain: An attacker first compromises a user account in a child domain within the same forest.
  1. Adding Parent Domain SIDs: The attacker adds the SID of a higher-privileged group from the parent domain (e.g., Enterprise Admins) to the SID History of the compromised user account.
  1. Forging a Trusted Ticket: The attacker then forges a Kerberos ticket that can be used to access resources in the parent domain, leveraging the added SIDs in the SID History.

This combination of SID-History Abuse and trusted ticket forgery allows the attacker to effectively compromise the entire forest, as long as they can gain a foothold in a single child domain.

Enumeration

Ok so assuming that we already have Access as Domain Admin and we would like to privilege escalate to Enterprise Admin.

To enumerate Domain Trusts we will be using PowerView Modules.

Get-DomainTrust

The output from Get-DomainTrust provides detailed information about the forest trust between dollarcorp.moneycorp.local and moneycorp.local.
Here's a breakdown of the fields:

Source and Target Names

  1. SourceName: The name of the source domain, which is dollarcorp.moneycorp.local.
  1. TargetName: The name of the target domain, which is moneycorp.local.

Trust Type and Attributes

  1. TrustType: The type of trust, which is WINDOWS_ACTIVE_DIRECTORY, indicating that it is a Windows Active Directory trust.
  1. TrustAttributes: The attributes of the trust, which are WITHIN_FOREST, indicating that the trust is within the same forest.

Trust Direction

  1. TrustDirection: The direction of the trust, which is Bidirectional, indicating that the trust is bidirectional, allowing both domains to access resources in each other.

netsh interface portproxy add v4tov4 listenport=8080 listenaddress=0.0.0.0 connectport=80 connectaddress=172.16.100.51

Let's go through the different flags used in the netsh interface portproxy add v4tov4 command:

  1. v4tov4: This specifies that the port forwarding is between IPv4 addresses. You can also use v6tov6 for IPv6 or v6tov4 and v4tov6 for mixed IPv4 and IPv6 scenarios.
  1. listenport:8080: This is the local port on the machine where the connection will be listened for. In your case, it's port 8080.
  1. listenaddress:0.0.0.0: This is the local IP address that the port will be listening on. Using 0.0.0.0 means it will listen on all available IP addresses on the machine.
  1. connectport:80: This is the port that the incoming connection on the local machine (port 8080) will be forwarded to. In your case, it's port 80.
  1. connectaddress:172.16.100.51: This is the IP address that the connection will be forwarded to. In your case, it's the IP address 172.16.100.51.

The purpose of this port forwarding configuration is to redirect any connections made to the local machine on port 8080 to the remote machine at IP address 172.16.100.51 on port 80.

The flags used while using Rubeus to forge a ticket with SID-History of Enterprise Admin are:

  1. /service: This flag specifies the service principal name (SPN) of the service that the ticket is being forged for. In this case, it is krbtgt/dollarcorp.moneycorp.local.
  1. /rc4: This flag specifies the RC4 key used to encrypt the ticket(dollarcorp.moneycorp.local). In this case, it is 11584b767b6316e5f48b35e029c32ade.
  1. /sid: This flag specifies the SID of the user or group that the ticket is being forged for. In this case, it is S-1-5-21-719815819-3726368948-3917688648.
  1. /sids: This flag specifies additional SIDs that should be included in the ticket(moneycorp.local). In this case, it is S-1-5-21-335606122-960912869-3279953914-519.
  1. /ldap: This flag specifies that the ticket should be forged using LDAP.
  1. /user: This flag specifies the username of the user that the ticket is being forged for. In this case, it is administrator.
  1. /nowrap: This flag specifies that the output should not be wrapped.
  1. /ptt: This flag specifies that the ticket should be printed to the console.

This command forges a ticket for the krbtgt/dollarcorp.moneycorp.local service using the RC4 key 11584b767b6316e5f48b35e029c32ade, the SID S-1-5-21-719815819-3726368948-3917688648, and the additional SID S-1-5-21-335606122-960912869-3279953914-519. The ticket is forged using LDAP and the username is administrator. The output is not wrapped and the ticket is printed to the console.
