    We encountered a persistent issue when attempting to extract credentials from the lsass_mem.dmp file using both Mimikatz and SafetyKatz. Despite loading the minidump successfully with sekurlsa::minidump, any attempt to run sekurlsa::logonpasswords failed with the error: ERROR kuhl_m_sekurlsa_acquireLSA ; Key import. This error typically indicates that Mimikatz is unable to import the necessary cryptographic material from the dump, which is essential to decrypt the LSA secrets.

    The root cause of this failure is likely tied to how the dump was generated. It appears the memory file was either created by a custom tool or in a protected environment where LSASS was running with LSA Protection (RunAsPPL). This protection encrypts sensitive memory regions, and unless the dump was taken with a tool capable of bypassing this protection, like Nanodump or kernel-assisted extraction, the result is a dump that is structurally intact but lacks accessible key material. Consequently, Mimikatz cannot resolve or import the keys needed to extract plaintext credentials or hashes.

    However, when we processed the same lsass_mem.dmp using pypykatz, it parsed the dump correctly and successfully enumerated the logon sessions. It did extract NTLM hashes and other relevant credential data. The key difference is that pypykatz uses a different parsing approach. It’s written in Python and does not rely on the same internal key import mechanism that Mimikatz uses. This makes it more tolerant of partial dumps or those missing LSA decryption keys. Instead of halting due to missing cryptographic material, it proceeds with parsing and attempts to extract whatever static credential data is present.

    PortScan

    Because the previous enumeration did not point to the valid domain we were supposed to move to, we should then look for another direction and this was the port scan on network 192.168.144.0/24.
    In a Windows-based environment where ICMP echo requests (ping) are typically blocked by Windows Defender Firewall, the classic nmap -sn ping scan becomes ineffective. To bypass that and reliably discover live hosts, especially during internal assessments where we're scanning from within the network (e.g., lateral movement scenarios), the best approach is to rely on TCP-based host discovery.

    nmap -n -Pn -p 445,135,139,3389,5985 --open 192.168.144.0/24

    Why this works:

    • n - disables DNS resolution, speeds things up, and avoids noise.
    • Pn - tells Nmap not to ping — it treats all hosts as online and probes them directly.
    • p 445 - targets SMB, which is almost always enabled on Windows endpoints.
    • -open - filters output to only show hosts that responded on that port (i.e., are alive).

    This strategy is effective because TCP 445 is typically open on domain-joined machines for file sharing, authentication, and other services. If the goal is to maximize discovery coverage, we can expand to other ports, we can do it as well, e.g: 21,22, 23, etc. It is possible to see on above screenshot that we discovered only one IP alive which was the 192.168.144.194.

    Now that we know that this host is alive, let’s add more flags to enumerate the services running on the just found open ports.

    nmap -Pn -sC -sV -p 445,135,139,3389,5985 192.168.144.197

Despite supplying the correct RC4 hash and specifying the domain, this approach failed with a KRB-ERROR (68): KDC_ERR_WRONG_REALM.     LSASS-based NTLM Session Forgery (No Kerberos)

    After that failure, we pivoted to using SafetyKatz and executed a pass-the-hash attack with the sekurlsa::pth command. This method worked. It launched a PowerShell process where our identity was forged in-memory as admin@secsyslog, using the same NTLM hash. Unlike Kerberos, this technique doesn’t involve reaching out to a domain controller for a TGT—it instead modifies the current process token and injects credentials locally. The OS treats that process as if it’s running under the impersonated user. We confirmed the identity was assumed correctly, but noticed that no Kerberos tickets were available (klist showed an empty cache). That was expected because this method does not automatically request a TGT; it simply crafts a new logon session with the provided hash.

    SafetyKatz.exe "privilege::debug" "sekurlsa::pth /user:admin /domain:SECSYSLOG /ntlm:fd9987e39827094aebac8233fefa519b /run:cmd" "exit"

Both methods failed with an error stating that the destination needed to be either in the TrustedHosts list or the connection needed to use HTTPS. Since we were targeting the host by its IP address and hadn’t modified the WinRM client policy, our session despite being forged as a domain admin was rejected by the remote machine due to the lack of explicit trust or secure transport. This confirmed that the issue wasn’t with the credentials, but with how WinRM handles authentication over IP in default configurations.

    Deep Dive on the Problem

    The technique that worked using SafetyKatz specifically the sekurlsa::pth command, is a local process-level token injection that enables us to impersonate a user by crafting a fake logon session directly within LSASS using only an NTLM hash. Rubeus, on the other hand, operates primarily through Kerberos ticket requests and manipulations. So no, Rubeus cannot do exactly what SafetyKatz (or Mimikatz) does with sekurlsa::pth. The two tools are solving related, but fundamentally different problems.

    That said, Rubeus does offer an equivalent outcome under the right conditions, but only if we can obtain a TGT. With that TGT, we can either:

    • Inject it directly into memory with /ptt (Pass-the-Ticket), or
    • Use /tgtdeleg or /s4u to pivot into constrained delegation scenarios.

    But if we only have the NTLM hash, and we are unable to reach the correct domain controller to get a TGT (as happened in our case), then Rubeus won't help because it relies on a successful Kerberos exchange. That’s the key limitation.

    When should we use SafetyKatz with sekurlsa::pth?

    We should use sekurlsa::pth when:

    • We have an NTLM hash, but no valid TGT.
    • We want to locally spawn a process that impersonates a target user.
    • We don't need to cross domain boundaries or request tickets.
    • We are already on a host where LSASS patching is permitted.
    • We need to authenticate to services (like SMB or PsExec) that accept NTLM authentication without needing a Kerberos ticket.

    This approach is excellent for local privilege escalation and scenarios where we have administrative control of a box and want to "become" another user, often a domain admin, for further action.

    When is Rubeus the better tool?

    Rubeus is the right choice when:

    • We have access to a domain controller in the same realm as the user we’re targeting.
    • We can communicate with the KDC and request tickets (TGT or TGS).
    • We want to manipulate or forge Kerberos tickets, including:
      • /asktgt for requesting tickets
      • /asreproast for brute-forceable accounts
      • /ptt for injecting a TGT or TGS into our session
      • /s4u for constrained delegation
    • We are operating in Kerberos-heavy environments, such as when accessing services like CIFS, LDAP, HTTP, or WinRM over SPNs.

    Rubeus is surgical and powerful, but it depends on working Kerberos infrastructure. When that infrastructure is unreachable or the realm is misaligned as we saw in our session Rubeus fails, and local hash injection is the more resilient path.

    To summarize: SafetyKatz’s pth technique is optimal when all we have is a hash and no TGT, and we want to impersonate locally. Rubeus excels when we can interact with Kerberos directly, request tickets, and perform operations like delegation, roasting, or ticket injection. Choosing between them comes down to the goal ticket-based operations vs. direct session impersonation and the constraints of the network path and authentication mechanisms.

    Finally, we turned to PsExec, passing the same target IP and launching PowerShell interactively. This succeeded immediately. The reason is simple: PsExec does not rely on WinRM or Kerberos. Instead, it uses SMB to copy a service executable to the target, executes it under NT AUTHORITY\SYSTEM, and tunnels the I/O back to our machine. Because our current process was already impersonating admin@secsyslog, and that account had administrative rights on the target, PsExec executed flawlessly and gave us a shell as NT AUTHORITY\SYSTEM.

    In the end, the difference came down to protocol behavior. Kerberos-based authentication via Rubeus failed due to a realm mismatch, and WinRM-based remoting was blocked by host trust restrictions. Meanwhile, the pass-the-hash via SafetyKatz created a valid local identity, and PsExec leveraged that identity using SMB, bypassing the need for ticket-based authentication altogether.

    .\PsExec64.exe \\192.168.144.197 -i -s cmd

    We encountered a persistent issue when attempting to extract credentials from LSASS using SafetyKatz and Mimikatz version 2.2.0. Despite having full administrative rights and even running as NT AUTHORITY\SYSTEM, every attempt to execute the sekurlsa::logonpasswords command resulted in the error below.

    ERROR kuhl_m_sekurlsa_acquireLSA ; Key import

    This error typically points to a failure in acquiring or interpreting the crypto material within LSASS memory. Given our privilege level and proper command syntax, initial assumptions about incorrect permissions or execution context were ruled out. We confirmed the issue persisted across both PowerShell and CMD environments, and that it was not related to clipboard injection or character mangling.

    After further investigation and validation, we discovered that this issue was caused by the version of Mimikatz/SaftyKatz being used. Mimikatz 2.2.0, while more recent, introduced changes in how it accesses and interprets LSASS memory structures. These changes, while designed for compatibility with modern Windows builds, are known to break support in specific scenarios particularly in environments where LSASS protection mechanisms (e.g., RunAsPPL) or security mitigations cause offset mismatches or disrupt expected memory layout.

    To confirm the source of the issue, we tested with an older version of Mimikatz specifically version 2.1.1, built in September 2018. Upon executing the exact same commands in the same context using this version, the operation succeeded immediately. Credential material was retrieved successfully from LSASS without any structural or cryptographic import failures.

    This confirms the root cause was not due to our environment or misuse of Mimikatz, but rather a compatibility regression in newer Mimikatz builds. The offset or module differences between LSASS and the newer mimikatz binary led to a parsing failure. Reverting to version 2.1.1 restored full compatibility and allowed us to continue with credential extraction as expected.

    This highlights a critical operational insight in post-exploitation workflows involving tools that parse undocumented memory structures, version pinning can be just as important as privilege level. When newer builds fail, using a previously stable version can often resolve the issue without altering the target environment or reducing OPSEC integrity.
